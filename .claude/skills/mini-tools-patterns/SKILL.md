# Patrones de conectores

Los 4 motores relacionales (Oracle, PostgreSQL, SQLite, SQL Server) se registran como drivers de `database/sql` en vez de usarse con sus SDKs nativos o con `sqlx`:

| Motor | Import | Driver name |
|---|---|---|
| Oracle | `github.com/sijms/go-ora/v2` | `"oracle"` |
| PostgreSQL | `github.com/jackc/pgx/v5/stdlib` | `"pgx"` |
| SQLite | `modernc.org/sqlite` | `"sqlite"` |
| SQL Server | `github.com/microsoft/go-mssqldb` | `"sqlserver"` |

Esto permite que `backend/db/connector.go` defina una única interfaz `Connector` y que `pool_manager.go` (`PoolManager`) mantenga un solo `map[connID]*sql.DB` sin importar el motor. Los 4 conectores están implementados (`sqlite.go`, `postgres.go`, `oracle.go`, `sqlserver.go`) — `ConnectorFor` ya no devuelve error para ningún `db_type` válido. Cada archivo de conector blank-importa su propio driver (`_ "modernc.org/sqlite"`, `_ "github.com/jackc/pgx/v5/stdlib"`, `_ "github.com/microsoft/go-mssqldb"`) — Oracle es la excepción porque `go_ora.BuildUrl` (no un blank import) es lo que se usa para construir el DSN, pero el driver igual se registra al importar el paquete. No asumir que otro paquete ya registró un driver — cada uno se registra a sí mismo, o los tests de ese paquete en aislamiento fallan con "unknown driver". Al añadir un motor nuevo:

1. Implementar `Connector` en `backend/db/<motor>.go` (builder de DSN a partir de la config de conexión, nunca aceptando un DSN crudo desde el frontend).
2. Registrar el pool en `pool_manager.go` — un pool por conexión activa, nunca reabrir por query.
3. Añadir las queries de metadata (`backend/db/metadata.go`) para poblar el árbol del sidebar y el autocomplete del editor.
4. Cerrar el pool explícitamente al eliminar o cambiar la conexión.

SQLite siempre en modo WAL al abrir (`PRAGMA journal_mode=WAL`). Postgres soporta el set completo de `sslmode` (disable/allow/prefer/require/verify-ca/verify-full) armando una URL `postgres://user:pass@host:port/db?sslmode=X` que pgx parsea. Oracle soporta los 4 modos de conexión del spec vía `params["mode"]`: `service_name`/`easy_connect` (misma forma DSN — Easy Connect ES literalmente `host:port/service`), `sid` (`go_ora.BuildUrl` con `service=""` + opción `SID=`), y `tns` (descriptor completo pegado de tnsnames.ora, pasado como opción `connStr=` — go-ora lo parsea y sus valores de host/protocolo/servicio pisan a los del DSN base). Ver `backend/db/oracle.go` para el detalle de cada modo.

**Costo de tamaño de binario de Oracle:** `go-ora` usa `crypto/tls` en su código core (soporte TCPS, no opcional, no excluible con build tags), y desde Go 1.24 eso arrastra el módulo FIPS 140-3 completo — agrega ~15MB al binario final. Esto ya forzó revisar el target de tamaño de <20MB a <35MB (ver [.claude/rules/technical.md](../../rules/technical.md) punto 8) — tenerlo en cuenta antes de añadir otra dependencia grande.

**SQL Server** (`backend/db/sqlserver.go`) usa el driver oficial de Microsoft `github.com/microsoft/go-mssqldb` (pure-Go, sin cgo, implementa `database/sql`) — se eligió por ser el más nativo y versátil de las opciones Go. Solo se importa el paquete base (auth SQL Server user/password); NO se importa el subpaquete `azuread` (Azure AD / Windows integrated), que arrastraría el SDK de Azure — Azure AD/SSPI quedan explícitamente fuera de alcance v1. DSN URL-shaped `sqlserver://user:pass@host:port/instance?database=db&encrypt=X&TrustServerCertificate=Y` (sirve tal cual para `export.RedactDSN`, mismo criterio que Postgres/Oracle). `encrypt` (disable/false/true/strict) default `disable` — mismo posture de red interna que el `InsecureIgnoreHostKey` de Oracle y el CORS permisivo. Una **instancia con nombre** (`instance`, va en el path del URL) es mutuamente excluyente con `port`: el SQL Server Browser resuelve el puerto, así que cuando hay instancia no se emite `:1433`. Metadata en `backend/db/metadata.go` (`fetchSQLServerMetadata`): columnas/PK vía `INFORMATION_SCHEMA`, FKs/rutinas/triggers vía `sys.*` (placeholders `@pN`, no `$N`/`:N`). Costo de binario: +3MB (41→44MB) — el módulo FIPS/`crypto/tls` que ya trae go-ora se comparte, así que no se paga de nuevo. **T-SQL ("variantes de PL/SQL"):** el separador de lotes `GO` (solo en su propia línea, opcionalmente con conteo `GO 5`) se maneja en `splitter.go` (`isLoneGoLine`) como frontera de statement — igual idea que el `/` de SQL*Plus, universal y seguro para los otros motores (una línea que sea solo `GO` no es SQL válido en ninguno). Un lote envuelto en `BEGIN...END` se mantiene como un solo statement (la lógica de nesting existente ya lo cubre), preservando el scope de variables `DECLARE @x`. **Limitación aceptada:** un script T-SQL con varios statements separados por `;` (sin `GO` ni `BEGIN/END`) se parte por `;` como los demás motores, así que variables de sesión declaradas antes de un `;` no sobreviven al siguiente statement — usar `GO` o envolver en `BEGIN...END` para forzar un lote único. `BEGIN TRANSACTION` (no `BEGIN`, que abre un bloque en T-SQL) en `executor.go`'s `transactionBeginStmt`. **No verificado contra una instancia SQL Server real** (no disponible en el entorno de desarrollo) — solo por inspección de código y un script efímero que valida BuildDSN/ParseDSN/RedactDSN roundtrip + el split de `GO`; mismo precedente de honestidad que Oracle/PL-SQL. DDL (`ddl_sqlserver.go`: reconstrucción de tabla desde `INFORMATION_SCHEMA` con quoting `[bracket]`, y `OBJECT_DEFINITION` para procedures/functions/triggers) y EXPLAIN (`sqlserver_plan.go`: `SET SHOWPLAN_ALL ON`, plan estimado sin ejecutar, `analyze` no distinguido en v1) también implementados sin verificar contra una instancia real.

# Patrones de ejecución de queries

**Split de statements** (`backend/query/splitter.go`, `SplitStatements`): divide un script en statements por `;` a nivel top-level, respetando comillas simples/dobles, comentarios `--`/`/* */`, dollar-quoting Postgres (`$$...$$` / `$tag$...$tag$`), y anidamiento `BEGIN...END` de Oracle (incluyendo la sección `DECLARE` y el cuerpo `IS/AS...BEGIN...END` de `CREATE [OR REPLACE] PROCEDURE/FUNCTION/TRIGGER/TYPE`). Reglas clave si tocás esto:
- `awaitingBegin` bloquea el split en la sección declare/parámetros (antes del primer `BEGIN`); `blockDepth` cuenta `BEGIN`(+1)/`END` bare(-1) una vez dentro. `END IF`/`END LOOP`/`END CASE` NO son un `END` bare — no decrementan `blockDepth` (van con su propio IF/LOOP/CASE, no con un BEGIN).
- Entrar a un dollar-quote (`$$`/`$tag$`) limpia `awaitingBegin` — si no, un cuerpo Postgres `AS $$ BEGIN ... END; $$ LANGUAGE plpgsql;` nunca cerraría (el `BEGIN` interno nunca se "ve" porque está protegido dentro del dollar-quote).
- **Limitación aceptada:** `CREATE PACKAGE BODY` con múltiples procedimientos miembro (múltiples pares BEGIN/END independientes) no se garantiza que divida bien — cada miembro re-entra en modo "awaitingBegin" que este tokenizer no rastrea. Bloques anónimos DECLARE/BEGIN y unidades CREATE PROCEDURE/FUNCTION/TRIGGER individuales sí funcionan. Ver `.claude/rules/technical.md` punto 7 (hand-rolled, no gramática completa).

`backend/query/detect.go` (`IsPLSQLBlock`) clasifica un statement como `KindSQL` o `KindPLSQLBlock` reusando la misma lógica de arranque que el splitter.

**Ejecución por statement, no por script:** `Executor.run` parte el `sqlText` con `SplitStatements` y corre cada uno por separado, emitiendo su propia secuencia `columns`/`rows`/`done` (o `error`/`cancelled`) bajo el mismo `queryID` pero con `Event.StatementIndex`/`TotalStatements` — así el frontend arma un result-tab por statement (`ResultTabs.tsx`) en vez de mezclar resultados de varios statements en un solo grid. Un error en un statement **no** detiene el script (siguen corriendo los demás, como sqlplus/mysql por default); una cancelación sí lo detiene por completo — el chequeo `ctx.Err()` al tope del loop hace `break`, no `continue` (un bug real de la primera versión: `continue` emitía un evento "cancelled" fantasma por cada statement restante en vez de simplemente dejar de correrlos).

**Bloques PL/SQL Oracle** (`backend/query/dbmsoutput.go`): un statement `KindPLSQLBlock` contra una conexión Oracle corre vía `runOraclePLSQLBlock`, que reserva una única `*sql.Conn` del pool (`pool.Conn(ctx)`) para `ENABLE`, ejecutar el bloque, y `GET_LINE` — todo en la MISMA sesión, porque el estado de `DBMS_OUTPUT` es por sesión; usar conexiones distintas del pool para cada paso simplemente no vería nada. Las líneas capturadas viajan en `Event.DBMSOutput`. Esta ruta no se pudo probar contra una instancia Oracle real (no disponible en el entorno de desarrollo) — solo por inspección de código.

Streaming de resultados: como los bindings de Wails v2 son petición/respuesta, `executor.go` nunca llama `runtime.EventsEmit` directamente — recibe un `query.EmitFunc` inyectado (`func(event string, data interface{})`) por quien lo construye. En producción (`app.go`) ese `EmitFunc` envuelve `runtime.EventsEmit(ctx, event, data)`; en tests es un stub que empuja a un channel. Esto es necesario porque `runtime.EventsEmit` exige un `context.Context` inyectado por el ciclo de vida real de Wails y llama `log.Fatalf` (mata el proceso) sin uno — no se puede testear el executor invocándolo directo. El **frontend genera el `queryID`** y llama `EventsOn(queryID, ...)` antes de invocar `ExecuteQuery(connID, queryID, sql)` — si el backend generara el ID habría una carrera entre el primer emit y la suscripción.

**Historial de queries:** separado de `EmitFunc` hay un segundo callback, `query.HistorySink` (`func(connID, sqlText, status string, rowsAffected, durationMs int64, errMsg string)`), invocado al final de cada statement (done/error/cancelled). En `app.go` apunta a `vault.Store.RecordQueryHistory`, que persiste en la tabla `query_history` (sin cifrar, a diferencia de `connections`). El paquete `query` no importa `vault` — se mantiene desacoplado igual que con `EmitFunc`.

Cancelación: cada `queryID` en curso tiene un `context.CancelFunc` registrado; `CancelQuery(queryID)` lo invoca y limpia el registro. El pool debe quedar sano después de cancelar (no cerrar la conexión subyacente, solo el statement en curso) — verificado en `backend/query/executor_test.go` con una CTE recursiva larga, y con un script multi-statement efímero (cancelar detiene antes del segundo statement).

# Patrones de metadata de schema

`backend/db/metadata.go` (`FetchSchemaMetadata`) unifica tablas/columnas/nullable/PK/FK en `db.SchemaMetadata`, una función por motor:
- SQLite: `PRAGMA table_info(tabla)` (columnas/nullable/PK) + `PRAGMA foreign_key_list(tabla)` (FK). El nombre de tabla se interpola con `%q` en el SQL del PRAGMA — SQLite no soporta bind params ahí — aceptable porque el nombre viene de `sqlite_master`, no de input de usuario.
- Postgres: 3 queries bulk (no N+1) sobre `information_schema.columns` + `table_constraints`/`key_column_usage` (PK) + `+ constraint_column_usage` (FK), agrupadas en Go por `schema.table`.
- Oracle: `user_tab_columns` (no `all_/dba_` — alcance del usuario conectado) + `user_constraints`/`user_cons_columns` para PK, y el patrón estándar de mapeo FK→PK por `position` para FK. No verificado contra una instancia Oracle real.

`App.GetSchemaMetadata(connID, forceRefresh)` cachea el resultado en memoria (`a.metadataCache`, un mapa protegido por mutex) — nunca se refresca solo, hace falta `forceRefresh=true` (el F5 del frontend). Verificado end-to-end con SQLite real y con un Postgres real en Docker (tablas con PK/FK/nullable reales).

# Patrones de CodeMirror (migración desde Monaco)

`monaco-editor` fue retirado por completo (todo `frontend/src/monaco/` y `MonacoTabbedEditor.tsx` eliminados) y reemplazado por CodeMirror 6 — mismo comportamiento funcional (vinculación de pestaña a conexión/lenguaje, linter con warnings bloqueantes/no-bloqueantes, atajos de teclado, autocompletado de tablas/columnas/keys), motor de edición distinto. `frontend/src/components/editor/CodeMirrorTabbedEditor.tsx` reemplaza `MonacoTabbedEditor.tsx`: mismo patrón "un editor compartido, un estado por pestaña" (`Map<tabId, EditorState>`, `view.setState(state)` al cambiar de pestaña activa), API distinta (`EditorState`/`Compartment` en vez de `ITextModel`/`setModel`).

**Diferencia clave con Monaco: sin providers globales/singleton.** Monaco registraba `completionProvider`/`hoverProvider` una sola vez por lenguaje y los alimentaba con holders mutables globales (`metadataStore.ts`, `activeDbTypeStore.ts`) porque los providers viven fuera del árbol de React. CodeMirror no tiene ese problema — cada pestaña tiene su propio `EditorState`, así que el schema/dialecto SQL y el tema van directo en las extensiones de ESE estado vía `Compartment` (`languageCompartment`/`themeCompartment` en `CodeMirrorTabbedEditor.tsx`), reconfigurados con `view.dispatch({effects: compartment.reconfigure(nuevaExtension)})` sin recrear el estado (no se pierde cursor/scroll/undo). Esto eliminó `metadataStore.ts`/`activeDbTypeStore.ts` sin reemplazo — `Workspace.tsx` pasa `dbType`/`schemaMetadata` como props normales en vez de empujarlos a un store global.

**Un detalle no obvio: `EditorState` es inmutable, así que un `EditorView.updateListener` reescribe `statesRef` en CADA update** (no solo cuando cambia el documento) — si solo se sincronizara en `docChanged`, un `Compartment.reconfigure()` (cambio de tema/schema) dejaría el Map con el estado viejo, y volver a esa pestaña más tarde revertiría el reconfigure. Mismo motivo por el que el reconfigure de tema recorre TODOS los estados cacheados (no solo el activo) al cambiar el tema global — si no, una pestaña de fondo mostraría un tema viejo un instante al volver a ella (un bug que Monaco no podía tener, porque su tema era un singleton verdadero — `monaco.editor.setTheme()` re-temaba todos los editores a la vez; CodeMirror no tiene ese concepto, cada `EditorState` es independiente).

**SQL vía `@codemirror/lang-sql`, no hand-rolled.** `frontend/src/codemirror/sqlSchema.ts` convierte `db.SchemaMetadata` a `SQLNamespace` (`schemaToNamespace`) y elige dialecto real por `dbType`: `PostgreSQL`/`SQLite`/`PLSQL` (Oracle — el dialecto PL/SQL real de la librería, mejor cobertura que la lista manual de `ORACLE_FUNCTIONS` que tenía `monaco/sqlLanguage.ts`). El autocompletado de tablas/columnas/alias/`schema.tabla` lo resuelve la librería sola — se retiró el análisis a mano (`monaco/sqlContext.ts`/`completionProvider.ts`). El hover SÍ se mantuvo hecho a mano (`sqlSchemaHover`, port de `monaco/hoverProvider.ts`) porque `@codemirror/lang-sql` no expone uno — reusa el mismo parser de `FROM`/`JOIN`/alias por regex que tenía `sqlContext.ts`, pero solo para resolver el hover, no el completado.

**Redis sigue sin paquete oficial** — `frontend/src/codemirror/redisLanguage.ts` es un port directo de `monaco/redisLanguage.ts`: mismo `REDIS_COMMANDS`/`firstArgIsKey`, tokenizer `StreamLanguage.define({token})` en vez de Monarch, `CompletionSource` en vez de `provideCompletionItems`, `hoverTooltip()` en vez de `registerHoverProvider`. Los snippets `${1:key}` de Monaco son sintaxis IDÉNTICA a la de `@codemirror/autocomplete`'s `snippetCompletion()` — no hubo que reescribir ningún template. `frontend/src/codemirror/redisKeysStore.ts` (movido tal cual desde `monaco/`) sigue siendo el único holder mutable global que sobrevivió — es intencional: `RedisKeyTree.tsx` ya gatea las escrituras por conexión-activa-de-la-pestaña, así que no hay cruce entre pestañas pese a ser global.

**Lint: todos los diagnósticos son `severity: 'warning'`, sin excepción.** `frontend/src/codemirror/lintAdapter.ts` adapta `lib/linter.ts`/`lib/redisLinter.ts` (sin cambios) a `Diagnostic[]` de `@codemirror/lint`. `LintWarning.blocking` NUNCA se mapeó a `severity: 'error'` — controla únicamente el diálogo de confirmación antes de ejecutar en `Workspace.tsx confirmAndRun` (para eso sigue leyendo `lib/linter.ts` directo), nunca el color del marcador en el editor. Confirmado leyendo el código de `MonacoTabbedEditor.tsx` antes de portar: sus markers eran siempre `monaco.MarkerSeverity.Warning`, sin excepción — mapear `blocking→error` habría sido una regresión visual no pedida.

**Temas vía `@uiw/codemirror-theme-*`, no el paquete base `@uiw/codemirror-themes`.** Ese paquete base es solo la infraestructura `createTheme()`/`Settings` — los presets reales (`githubLight`/`githubDark`, `dracula`, `nord`, `material`, `solarizedLight`/`solarizedDark`, `vscodeDark`, `tokyoNight`) están en paquetes por-tema separados. `frontend/src/codemirror/themes.ts` registra 9 presets + `"auto"` (sigue el toggle claro/oscuro de la app, resuelto en `resolveEditorTheme`), persistido en `settings.editor_theme` (migración v9, `SetEditorTheme` en `app.go`/`settings_repo.go`). El toggle claro/oscuro de la app (`useTheme.ts`) ya NO toca el editor — antes llamaba `monaco.editor.setTheme()` (singleton global), ahora `CodeMirrorTabbedEditor` resuelve su propio tema por Compartment a partir del prop `appTheme` + `editorThemeId`, completamente desacoplado.

**Bug de empaquetado real: `@uiw/codemirror-theme-*` requiere `@babel/runtime` sin declararlo como dependencia.** El build de Vite falla con `Rollup failed to resolve import "@babel/runtime/helpers/extends"` si no está instalado — los paquetes `@uiw/codemirror-theme-*` se compilan con `@babel/plugin-transform-runtime` pero no listan `@babel/runtime` en su `package.json`. Fix: `pnpm add @babel/runtime` explícito en `frontend/package.json`. Si se agrega un preset `@uiw/codemirror-theme-*` nuevo, este ya cubre la dependencia — no hace falta repetir el fix.

CodeMirror 6 resultó **más liviano** que Monaco recortado a SQL pese a sumar autocompletado por esquema, dialectos SQL reales, lint y search: el binario bajó de 44MB a 41MB (ver [.claude/rules/technical.md](../../rules/technical.md) punto 8). No se agregó minimap (CodeMirror no tiene un paquete oficial equivalente) — diferencia visual menor, no funcional, no reportada como bloqueante.

# Autocompletado SQL: `@codemirror/lang-sql` no distingue cláusula (post-lanzamiento)

**Bug real reportado por el usuario contra un Oracle real** (schema con decenas de tablas/sinónimos `exam_*`): tipear `WHERE exam_` sugería, mezclados, `exmaple` de la tabla del `FROM` junto con `exam_PROV`, `exmaple_H`, `exmaple_PASO` y otras tablas/sinónimos del schema completo que nada tienen que ver con la consulta — la sugerencia correcta aparecía primero pero enterrada entre ruido.

**Causa raíz confirmada leyendo el fuente compilado** (`node_modules/@codemirror/lang-sql/dist/index.js`, `completeFromSchema`): la completion schema-aware de la librería NO tiene noción de cláusula — en una posición "bare" (sin punto), matchea el prefijo tipeado contra TODO el namespace de nivel superior (todas las tablas + `defaultTable`/`defaultSchema` si están configurados), sin importar si el cursor está después de `FROM` o después de `WHERE`. `defaultTable`/`defaultSchema` (`SQLConfig`) además son **aditivos, nunca exclusivos** — agregan columnas al nivel superior, no restringen qué más aparece. No hay ninguna opción nativa para "solo columnas de esta tabla, nada más".

**Fix (`frontend/src/codemirror/sqlSchema.ts`):** se dejó de pasar `schema` a `sql({...})` — confirmado en el mismo fuente que `schemaCompletion(config)` retorna `[]` cuando `config.schema` es falsy, así que la librería no registra NINGUNA fuente de completado basada en schema (su `keywordCompletionSource` sigue intacta, no depende de `schema`). En su lugar, `schemaAwareCompletionSource` toma control total del completado basado en schema: reintroduce un detector de cláusula liviano (`detectClause`, mismo enfoque que el retirado `monaco/sqlContext.ts` — último keyword relevante antes del cursor gana: `FROM`/`JOIN`/`INTO`/`UPDATE` → `'from'`, `SELECT`/`WHERE`/`SET`/`ON`/`AND`/`OR`/`GROUP`/`ORDER`/`HAVING`/`BY` → `'column'`) y con eso decide:
- Posición con punto (`alias.`/`tabla.`/`schema.`): resuelve vía el mismo `extractTableRefs` que ya usaba el hover — solo columnas/tablas de ESE alcance.
- Cláusula `'from'`: nombres de tabla + schema (sin cambios de comportamiento).
- Cláusula `'column'`: SOLO las columnas de las tablas referenciadas en el `FROM`/`JOIN` del statement — el fix real. Si ningún ref resuelve a una tabla conocida (consulta nueva sin `FROM` aún), cae a columnas de TODAS las tablas — nunca sugiere menos que antes.

**Edge case heredado del retirado `sqlContext.ts` y re-portado:** `INSERT INTO tabla (col1, col2|` — el último keyword visto es `INTO` (clasificaría como `'from'`, tabla), pero un paréntesis sin cerrar después de la referencia de tabla significa que ya se está en la lista de columnas. `detectClause` cuenta paréntesis abiertos/cerrados desde el final de `INTO` y fuerza `'column'` si hay un `(` sin cerrar.

**Verificado con un script Node standalone** (mismo patrón que `connStringParser.ts`/el retirado `sqlContext.ts` — mirror en JS plano de `detectClause`, sin test nuevo) cubriendo 13 casos: el de la screenshot original, `FROM` bare, lista de columnas en `SELECT`, `AND`/`ON`/`SET`, `INSERT INTO` con y sin paréntesis abierto, múltiples tablas en `FROM`, y un script multi-statement (el detector encuentra el último keyword más cercano al cursor sin importar que cruce un `;` de statement anterior, porque siempre se queda con el match de mayor índice). Un solo caso quedó como limitación aceptada y documentada: dentro de `VALUES (|` ninguna clasificación es realmente correcta (no es tabla ni columna) — `'column'` es la menos mala de las dos, no amerita más lógica especial.

**No se tocó `sqlSchemaHover`** (el hover del cursor sobre tabla/columna) — sigue usando su propia resolución de alias/punto independiente, sin relación con este cambio salvo compartir `extractTableRefs`.

# Patrones de export

`backend/export/` tiene un archivo por formato/motor, todos funciones puras (reciben `columns`/`rows` o un `*sql.DB` + nombre de tabla, devuelven texto/escriben un archivo, sin tocar `vault` ni `App`). `app.go` es quien sabe pedir el diálogo (`runtime.SaveFileDialog`) y decidir el nombre sugerido — mismo patrón que `BackupVault`.

**DDL por motor:** SQLite lee `sqlite_master.sql` tal cual (sin reconstrucción, siempre exacto). Postgres reconstruye desde `information_schema` — sí incluye PK y FK (ver `ddl_postgres.go`), NO incluye check constraints, unique no-PK, tipos custom ni particionado; si se agrega alguna de esas, extender ahí, no crear un archivo nuevo. Oracle usa `DBMS_METADATA.GET_DDL('TABLE', tabla)` — mucho más simple que reconstruir a mano porque Oracle ya lo expone, pero no verificado contra una instancia real.

**`ExportResult` no busca resultados por `queryID`:** el executor no retiene nada después de emitir (streaming puro, Fase 5), así que exportar significa que el frontend reenvía los `columns`/`rows` que ya tiene en memoria — esto es una desviación intencional del plan original (`ExportResultToFile(queryID, format, destPath)`). Si en algún momento se necesita exportar un resultado que el usuario ya cerró/perdió del frontend, la solución sería agregar retención server-side por `queryID` con un TTL, no volver a tratar de buscar algo que nunca se guardó.

**Redacción de DSN** (`connection_export.go`, `RedactDSN`): funciona para los 3 motores porque los 3 DSN son URLs (`scheme://user:pass@host/...` — ver `backend/db/{sqlite,postgres,oracle}.go`). Si algún motor nuevo no genera un DSN URL-shaped, esta función no le sirve tal cual.

**"Copiar como INSERT" es frontend-only** (`frontend/src/lib/sqlGenerate.ts`) — formateo de texto + `navigator.clipboard`, sin I/O de archivo, así que no hay razón para un viaje a Go. Desviación del plan original (`backend/export/sqlgen.go`).

**Sort de grid = requery, no sort en cliente:** `Event.SQLText` (agregado en Fase 7) lleva el texto exacto del statement en el evento `"columns"` — así `Workspace.tsx` puede envolver ESE statement puntual en `SELECT * FROM (...) AS mt_sort ORDER BY "col" ASC|DESC` al hacer click en un header, sin tener que duplicar `splitter.go` en TypeScript para saber dónde empieza/termina cada statement de un script multi-statement.

# Patrones de EXPLAIN PLAN

Un `PlanNode`/`Plan` común (`backend/explain/tree.go`) para 3 formatos nativos totalmente distintos — al agregar cualquier lógica que dependa de la forma del plan (ej. un nuevo tipo de warning visual), hacerlo sobre el árbol común, no sobre el formato crudo de un motor específico.

**Ojo con asumir el texto exacto de `EXPLAIN QUERY PLAN` de SQLite sin verificar en vivo:** la primera versión de `isSQLiteFullScan` buscaba el substring `"SCAN TABLE"`, basado en documentación/memoria — pero SQLite moderno solo escribe `"SCAN people"` (sin "TABLE"), así que la detección de full scan nunca disparaba, silenciosamente, hasta que se probó contra una base real con una tabla sin índice. La lección: para cualquier detección basada en parseo de texto de una herramienta externa (EXPLAIN, DDL, etc.), verificar el string real que devuelve la versión instalada, no solo lo que dice la documentación o lo que uno recuerda. La detección correcta es prefijo `"SCAN"` excluyendo `"USING INDEX"`/`"USING COVERING INDEX"` (esas son scans de un índice, no de la tabla completa).

**Oracle EXPLAIN PLAN necesita una `*sql.Conn` reservada + `statement_id` único**, igual que `DBMS_OUTPUT` en `backend/query/dbmsoutput.go` — `EXPLAIN PLAN FOR` escribe a la tabla compartida `PLAN_TABLE`, y sin un `statement_id` único por llamada, EXPLAINs concurrentes de otras sesiones podrían pisarse. No verificado contra una instancia real.

**Postgres `ANALYZE` ejecuta la query de verdad** (no es gratis, especialmente en UPDATE/DELETE — aunque EXPLAIN ANALYZE de por sí ya hace rollback de cualquier escritura, sigue corriendo el plan completo). Sin `ANALYZE`, `Plan.DurationMs` y `PlanNode.ActualTimeMs` quedan en 0 (son solo estimaciones del planner, nunca tiempos reales) — verificado explícitamente con ambos casos contra una base real.

# Patrones de generación de CLAUDE.md (Fase 9)

**El CLAUDE.md generado documenta la base de datos conectada, no mini-tools.** Decisión explícita del usuario (vía `AskUserQuestion`, tras señalar la ambigüedad del spec original) — `backend/claudemd/` produce documentación del *schema* (tablas/columnas/FKs/convenciones SQL del motor) para que Claude Code, trabajando en el proyecto SQL de un tercero, tenga contexto de la base contra la que corre. Si en algún momento se pide lo contrario ("documentar mini-tools mismo"), es un paquete/generador completamente distinto — no extender este.

**`Generate` vs `Regenerate`:** `Generate(dir, info)` es el trigger automático (silencioso, no pisa un `CLAUDE.md` existente, retorna `wrote bool` para que el caller sepa si hizo algo). `Regenerate(dir, info)` es la única función que sobreescribe — se usa solo desde la acción explícita del usuario ("Regenerar CLAUDE.md" en la toolbar). Nunca llamar `Regenerate` desde un trigger automático (abrir/guardar archivo) — perdería cualquier ajuste manual que el usuario le haya hecho al CLAUDE.md generado.

**`ProjectInfo` nunca lleva DSN/password** — solo `ConnectionName`/`DBType`/`*db.SchemaMetadata`, los mismos datos que ya son seguros de mostrar en el frontend (`vault.ConnectionSummary` tampoco expone el DSN). Si se agrega un campo nuevo a `ProjectInfo`, verificar que siga siendo así antes de que los templates lo rendericen a un archivo que termina commiteado en el repo del usuario.

**`connID` explícito en vez de "conexión activa" server-side:** `App.GenerateProjectDocs(projectRootPath, connID)`/`RegenerateProjectDocs(projectRootPath, connID)` desvían del plan original (un solo argumento) porque `App` no tiene estado de "conexión seleccionada" — eso vive solo en `Workspace.tsx` (`selected`). Cualquier método nuevo de `App` que necesite "la conexión con la que se está trabajando" debe seguir este mismo patrón (recibir `connID` explícito), no inventar un estado de sesión server-side nuevo.

**No hay `path.dirname` en el frontend** (contexto Vite/browser, sin Node): `dirName()` en `Workspace.tsx` es un split manual por `/`/`\` — mismo patrón que el `fileTitle()` ya existente. Cualquier helper de paths nuevo en frontend debe seguir esa misma forma manual, no importar `path` de Node (no está disponible en el bundle del browser).

**Post-lanzamiento: `GenerateProjectDocs`/`RegenerateProjectDocs` ahora reciben un tercer argumento `schema string`** (`app.go`, `claudemd.ProjectInfo.Schema`) — el schema activo del dropdown de `Workspace.tsx` (`activeSchema ?? ''`). Cuando no es `""`, `buildClaudeMDInfo` filtra `meta.Tables` a `t.Schema == schema` antes de pasarlo a los templates, y los 4 templates (`renderClaudeMD`/`renderSchemaSpec`/`renderSkill`) imprimen una nota explícita de alcance ("este documento solo cubre el esquema X") para que el CLAUDE.md generado nunca mienta sobre qué documenta. `regenerateProjectDocs()` en `Workspace.tsx` también agrega un `window.confirm()` (sobreescribe archivos existentes) y un estado `regeneratingDocs` para deshabilitar el botón mientras corre — antes no tenía ninguna de las dos cosas. Cualquier caller nuevo de estas dos funciones debe pasar `activeSchema ?? ''`, nunca omitir el tercer argumento ni hardcodear `''`.

# Patrones de theming, lazy-load y seguridad (Fase 10)

**`GetSettings`/`SetTheme` son la única excepción al "todo método pasa por `requireUnlocked`":** el tema es un dato no sensible (nunca DSN, nunca filas), y gatearlo detrás de la clave maestra solo produciría un flash de tema incorrecto en la pantalla de desbloqueo. Cualquier preferencia futura igual de no-sensible (tamaño de fuente, layout) debería seguir este mismo patrón — pero cualquier cosa que toque `encrypted_dsn` o datos de conexión SIEMPRE debe seguir gateada, sin excepción.

**El tema vive en un solo hook (`useTheme.ts`), llamado una única vez en `App.tsx`, nunca en `UnlockScreen`/`Workspace` directamente** — evita fetches duplicados de `GetSettings` y mantiene el estado consistente entre ambas pantallas (que de todas formas nunca coexisten, por el flujo `isInitialized`/`unlocked`). Se pasa hacia abajo por props (`theme`, `onToggleTheme`) — este proyecto nunca terminó usando Zustand pese a que el plan original lo mencionaba para un `uiStore`; todo el estado del frontend es `useState`/props (ver `Workspace.tsx`), así que cualquier estado nuevo compartido entre pocos componentes debería seguir ese mismo patrón antes de introducir una librería de estado global.

**(Retirado post-migración a CodeMirror — ver "Patrones de CodeMirror" arriba.)** Esta nota describía el tema de Monaco como singleton global (`monaco.editor.setTheme()` re-temaba todos los editores a la vez). Ya no aplica: `useTheme.ts` no toca el editor en absoluto, y el tema del editor es un `Compartment` por pestaña (`CodeMirrorTabbedEditor.tsx`), resuelto desde `editorThemeId` + el prop `appTheme`.

**Conversión mecánica a light/dark:** el frontend se escribió originalmente dark-only (clases `bg-neutral-950` etc. sin condicional). La conversión a pares `light dark:dark` se hizo con un script Python de un solo uso (no commiteado) usando una escala de grises invertida simétrica alrededor de 500 (`50↔950, 100↔900, ..., 500` sin cambio), aplicado uniformemente sobre `bg|border|text|divide|ring|placeholder-neutral-*` respetando prefijos de variante (`hover:`, `focus:`, `odd:`, `even:`) y sufijos de opacidad (`/40`, `/60`). Los colores de acento (`text-red-400`, `text-emerald-400`, `text-amber-400`) se ajustaron a mano con la misma idea porque son pocos casos y con excepciones (los botones sólidos con texto blanco como `bg-emerald-700` se dejaron igual en ambos temas, ya tienen contraste suficiente). Cualquier componente nuevo debe escribir directamente los pares `claseLight dark:claseDark` desde el principio, no clases dark-only a corregir después.

**Candidatos correctos para `React.lazy`: solo lo que está gateado por un booleano de "mostrar/ocultar", nunca lo que se renderiza siempre.** `ConnectionDialog`/`ExplainPlanPanel` (detrás de `showDialog`/`showExplain`) sí se lazy-cargaron: son chunks separados (~11KB/~2KB) fuera del bundle inicial. `ExportMenu` NO se lazy-cargó porque se renderiza incondicionalmente en el toolbar de resultados — envolverlo en `Suspense` solo agregaría un fallback sin ahorrar nada. El editor (`CodeMirrorTabbedEditor.tsx`) tampoco es candidato — el workspace lo necesita de inmediato. Antes de envolver un componente nuevo en `lazy()`, confirmar que de verdad está detrás de un flag condicional, no solo que "se ve pesado".

**`vaultgate.Gate.Lock()` puede quedar como código muerto si nada lo invoca** — estaba definido desde la Fase 2 pero nunca se llamaba desde ningún lado, así que la clave derivada vivía en memoria sin zeroear durante toda la sesión, y solo se liberaba (sin zeroear) cuando el proceso terminaba. `app.go`'s `shutdown()` ahora lo invoca. Si en el futuro se agrega un botón "Bloquear"/"Logout" explícito, debe llamar `a.gate.Lock()` también ahí, no solo confiar en `shutdown()`.

**Nunca zeroear una clave después de `gate.Set(key)`** — `Gate.Set` guarda el `[]byte` por referencia, no por copia, así que `mtcrypto.Zero(key)` después de un `Set(key)` exitoso zerearía la clave real que el gate acaba de guardar (rompiendo el vault en memoria). Solo zeroear una clave derivada en el path donde **nunca** llega a `gate.Set` — p.ej. el path de contraseña incorrecta en `Store.Unlock`, donde la clave se deriva, se prueba, y se descarta.

**Gotcha real de esta fase: `appdata.Dir()` no tiene override para tests/scripts, y un script efímero sin sandboxear escribe en la ruta real del usuario (`~/Library/Application Support/mini-tools`), no en un temp dir.** Un primer intento de verificar el fix de zero-out llamó `vault.Open()` directo y terminó inicializando el vault real de la máquina con una contraseña de prueba (recuperado borrando `vault.db`/`salt.bin` reales — sin pérdida de datos porque el vault nunca había sido inicializado antes, así que no podía tener conexiones guardadas). La forma correcta: `os.UserConfigDir()` en Darwin respeta `$HOME`, así que cualquier script efímero que toque `vault.Open()`/`appdata.Dir()` (o cualquier código que dependa de ellos transitivamente) debe correr como `HOME=$(mktemp -d) go run ./tmp_xxx`, nunca sin ese override. Esto aplica a **cualquier** verificación futura de vault/crypto/gate, no solo a la de esta fase.

# Parser de connection strings pegadas (`frontend/src/lib/connStringParser.ts`)

`ConnectionDialog.tsx` tiene un textarea "Pegar connection string" que corre `parseConnectionString(raw)` en cada `onChange` y, si detecta el formato, autocompleta `dbType`/`oracleMode`/`params` de una — pensado para copiar-pegar directo de un `.env`. Cubre: URLs `postgres://`/`postgresql://`, forma libpq `key=value` (`host=... user=... dbname=...`), rutas SQLite sueltas o con scheme `sqlite://`/`file://`, y 4 formas de Oracle (`oracle://` estilo go-ora, Easy Connect `user/pass@host:port/service` o `:sid`, JDBC thin `jdbc:oracle:thin:...` en sus variantes con/sin credenciales, y un descriptor TNS `(DESCRIPTION=...)` suelto o prefijado con `user/pass@`). Es best-effort: si no reconoce el formato retorna `null` y el diálogo queda exactamente como antes (completable a mano) — no se pierde nada por intentarlo.

**Bug real encontrado y corregido durante la verificación:** `stripEnvWrapper` (que quita un prefijo `KEY=` estilo `.env`) originalmente aceptaba cualquier identificador `[A-Za-z_][A-Za-z0-9_]*` antes del `=`, incluyendo minúsculas — lo que le hacía comerse el `host=` de una connection string libpq pegada directo (`host=localhost port=5432 ...`), tratándola por error como si fuera un wrapper de env var y perdiendo el primer campo. Corregido restringiendo el regex a `[A-Z][A-Z0-9_]*` (SCREAMING_SNAKE_CASE, la convención real de `.env`) — una key en minúscula como `host=` nunca es un wrapper de env var, es parte de la connection string misma. Verificado con un script standalone de Node (mirror en JS plano del parser, sin tests nuevos — mismo patrón de verificación efímera del proyecto) cubriendo las 4 formas de Postgres/SQLite/Oracle antes descritas más el caso ambiguo que disparó el bug.

**Por qué no usar una librería de parsing de DSN:** mismo argumento que el splitter de SQL (`backend/query/splitter.go`) — ninguna librería cubre bien Postgres + Oracle (con sus 4 formatos) + SQLite a la vez, y la superficie real a cubrir es acotada. Si se necesita soportar un formato nuevo, agregar otra rama de detección en `parseConnectionString` en vez de traer una dependencia.

# Bug real: `SELECT *` bloqueaba la ejecución (linter demasiado estricto)

`lintSQL` (`frontend/src/lib/linter.ts`) marcaba `SELECT *` con la misma severidad que `UPDATE`/`DELETE` sin `WHERE`, y `confirmAndRun` (`Workspace.tsx`) mostraba un `window.confirm()` **bloqueante** ante cualquier warning — incluido `SELECT *`. Un usuario reportó "no me deja ejecutar ni un SELECT * simple", que en realidad era el diálogo nativo de confirmación no siendo percibido como tal (fácil de confundir con "la app no responde", sobre todo en un webview embebido donde un `window.confirm()` puede no destacar visualmente).

`SELECT *` es una preferencia de estilo (puede traer columnas de más), no un riesgo real de pérdida de datos como `UPDATE`/`DELETE` sin `WHERE` — no ameritaba bloquear un query de solo lectura. Se agregó un campo `blocking: boolean` a `LintWarning`: `false` para `SELECT *` (sigue marcado con subrayado en el editor — hoy vía `frontend/src/codemirror/lintAdapter.ts`'s `lintExtension`, que tampoco filtra por `blocking` — el hint visual se mantiene), `true` para `UPDATE`/`DELETE` sin `WHERE` (sigue bloqueando con el diálogo de confirmación). `confirmAndRun` ahora filtra `lintSQL(sqlText).filter(w => w.blocking)` antes de decidir si confirma.

**Lección para cualquier warning nuevo que se agregue al linter:** por defecto `blocking: false` (solo marcador visual) — solo poner `blocking: true` si el warning cubre algo genuinamente destructivo/irreversible (pérdida de datos), no por preferencia de estilo o performance.

# Transacciones explícitas / auto-commit toggle (post-lanzamiento)

`Executor.BeginTransaction`/`CommitTransaction`/`RollbackTransaction` (`backend/query/executor.go`) reservan un `*sql.Conn` crudo por `connID`, nunca el `*sql.Tx` de Go — la razón es que un `COMMIT`/`ROLLBACK` que el usuario escriba a mano (o dentro de un bloque PL/SQL) tiene que funcionar exactamente como en psql/sqlplus sobre la misma sesión, y `*sql.Tx` se pelearía con eso. Mientras hay una conexión reservada para un `connID`, TODO statement que le llegue (`runQuery`/`runExec`/`runPLSQLBlock`) se enruta a esa conexión en vez del pool — ver la interfaz `queryExecer`. El "auto-commit está activado de nuevo" no es un booleano separado — es simplemente "ya no hay conexión reservada para este `connID`".

**Si se agrega un nuevo caller que ejecuta SQL sobre un `connID`** (fuera de `Executor.run`), hay que acordarse de chequear `e.txConn(connID)` primero y usar esa conexión si existe — de lo contrario ese caller ignoraría silenciosamente una transacción abierta por el usuario y correría contra el pool compartido, lo cual rompe la garantía de "todo dentro de la transacción ve la misma sesión". `runPLSQLBlock` es el ejemplo real de esto: antes de esta feature siempre reservaba su propia conexión (necesaria para `DBMS_OUTPUT`); ahora primero pregunta si ya hay una reservada por una transacción abierta y la reutiliza en ese caso, para que un bloque PL/SQL con su propio `COMMIT` afecte la MISMA transacción que las UI-buttons de Commit/Rollback están rastreando.

**Nunca dejar una conexión reservada sin liberar** — `DisconnectConnection`/`DeleteConnection` (`app.go`) y `shutdown()` (vía `Executor.RollbackAll`) hacen rollback best-effort de cualquier transacción abierta ANTES de cerrar el pool/la app. `sql.DB.Close()` no alcanza una conexión que un caller ya sacó del pool con `Conn()` y nunca devolvió — sin este cleanup, esa conexión se filtra.

**Verificado contra un Postgres real en Docker** (aislado, nunca el `local-postgres-1` del usuario — mismo patrón de siempre): aislamiento de una transacción abierta vs. otra conexión, 100 statements committeados recién al final, rollback descartando todo, doble-begin fallando limpio, y el caso de un `COMMIT` inline dentro del script (aplica inmediato, la conexión sigue reservada, el Rollback posterior de la UI es un no-op seguro).

**UX del toolbar (post-lanzamiento):** `Workspace.tsx` tiene un toolbar de 2 filas — fila de contexto (conexión/schema/auto-commit) y fila de acciones (archivo/query/utilidad), separadas por un borde, cada una con sus propios clusters divididos por `<Divider />`. El control de auto-commit es un checkbox (`checked={!txOpen}`), no un botón de texto — más estándar y reconocible que un botón "Auto-commit: ON" que hay que leer para entender el estado actual. Los botones Commit/Rollback están **siempre visibles** cuando hay una conexión seleccionada (no aparecen/desaparecen condicionalmente) y se deshabilitan con `disabled={!txOpen}` — evita que el layout salte cada vez que se abre/cierra una transacción, un anti-patrón de UX que el diseño original tenía. Mismo criterio aplica a cualquier control futuro cuya disponibilidad dependa de un estado transitorio: preferir siempre-visible-pero-disabled sobre aparece/desaparece, salvo que el control no tenga sentido en absoluto sin ese estado (p.ej. el selector de schema, que no tiene nada que mostrar si `metadata` no trajo ningún schema).

# Editar conexión / restaurar pestañas (post-lanzamiento)

**`Connector.ParseDSN` siempre incluye el password — la redacción es responsabilidad de quien lo llama, no de la función.** `app.go`'s `GetConnectionForEdit` borra `password` del mapa antes de devolverlo al frontend; `UpdateConnection` en cambio SÍ necesita el password real internamente (para hacer merge cuando el usuario deja el campo en blanco), así que llama `ParseDSN` de nuevo sobre el DSN actual y toma `password` de ahí, nunca del mapa ya redactado. Si se agrega un nuevo caller de `ParseDSN`, hay que decidir explícitamente de qué lado de esa línea cae — nunca asumir que ya viene limpio.

**"Dejar el campo en blanco" en el diálogo de edición significa "mantener lo que ya había", no "borrarlo"** — porque el frontend nunca tuvo el password real para empezar (`GetConnectionForEdit` lo redacta). Cualquier campo nuevo que en el futuro se vuelva sensible (y por lo tanto también se redacte al editar) necesita el mismo patrón de merge en el `Update*` correspondiente, o un guardado en blanco lo borraría silenciosamente.

**El toggle de restauración de pestañas tiene una carrera real que hay que respetar:** `Workspace.tsx` arranca con un tab scratch (`tabs = [newScratchTab()]`) antes de que la restauración asíncrona (`GetSettings` seguido de `OpenSQLFilePath` por cada path) termine. Un efecto que persiste `tabs` cada vez que cambia, si corre antes de que la restauración termine, sobreescribe `open_tabs` en disco con la lista vacía del scratch tab — borrando lo que se acababa de leer. `hasRestoredRef` (un `useRef`, no `useState` — no necesita re-render) es el guard: el efecto de persistencia es un no-op hasta que la restauración marca `hasRestoredRef.current = true` en su bloque final. Cualquier feature nueva de "cargar algo persistido al arrancar y volver a persistirlo cuando cambia" necesita este mismo guard, o corre el mismo riesgo de auto-borrarse en el primer render.

**Gotcha real de esta sesión: un separador pensado como texto de escape terminó escrito como byte crudo en el archivo fuente.** Al diseñar la key de un efecto para que no dispare en cada tecla tipeada, se buscó un separador poco probable de aparecer en un path real — y ese carácter terminó incrustado literalmente en `Workspace.tsx` en vez de como texto imprimible, dejando al archivo como binario para herramientas normales de texto (`file` ya no lo veía como texto plano, `grep` dejaba de encontrar coincidencias sin avisar del problema). Se detectó porque un `Edit` posterior sobre esa misma línea falló al no encontrar el string exacto, y se confirmó leyendo el archivo con `python3` en modo binario, buscando bytes fuera del rango imprimible. Se corrigió reemplazando ese byte por un espacio simple. Lección: para un separador "que no va a colisionar con datos reales", usar siempre algo imprimible y obviamente inofensivo (un espacio, una barra) — el riesgo de terminar escribiendo el carácter crudo en vez del texto que lo representa no vale la pena para un caso de bajísimo impacto si llegara a colisionar.

# Autocomplete consciente del contexto SQL (post-lanzamiento, retirado en la migración a CodeMirror)

**Retirado:** `sqlContext.ts`/`completionProvider.ts` (el análisis de contexto para completado) ya no existen — `@codemirror/lang-sql` resuelve FROM/JOIN/alias/`schema.tabla` nativamente (ver "Patrones de CodeMirror" arriba). El parser de `extractTableRefs`/`ALIAS_STOPWORDS` que describe esta sección SÍ sobrevivió, pero solo para el hover (`frontend/src/codemirror/sqlSchema.ts`'s `sqlSchemaHover`) — la librería no trae uno propio. Queda como historia de por qué el parser tiene la forma que tiene.

**`frontend/src/monaco/sqlContext.ts`** (nuevo, sin dependencia de Monaco — parseable/testeable standalone, mismo patrón que `linter.ts`/`connStringParser.ts`) analiza el statement donde está el cursor y devuelve: `clause` (`'from'` tras FROM/JOIN/INTO/UPDATE → sugerir tablas; `'column'` tras SELECT/WHERE/SET/ON/AND/OR/GROUP·ORDER BY → sugerir columnas; `'other'` para todo lo demás → cae al comportamiento viejo de sugerir todo, nunca sugiere *menos* que antes), `tableRefs` (cada tabla referenciada en FROM/JOIN/INTO/UPDATE del statement completo, con su alias si tiene), y `dotPrefix` (el identificador justo antes de un "." al cursor, para resolver "alias.", "tabla.", o "esquema." a sugerencias acotadas). `completionProvider.ts` lo usa así: `dotPrefix` resuelto tiene prioridad total (solo columnas de esa tabla, o solo tablas de ese esquema); si no hay dot, `clause==='column'` acota las columnas sugeridas a las tablas realmente referenciadas en el FROM/JOIN de *ese* statement (no todas las tablas del schema activo) — cuando `tableRefs` está vacío (statement sin FROM parseable todavía) cae a sugerir columnas de todas las tablas, mismo criterio de "nunca sugerir menos que el comportamiento anterior".

**Deliberadamente sin parser real, mismo argumento que `backend/query/splitter.go` vs. una librería completa:** el split de statements es ingenuo (split por `;` de nivel superior, sin conciencia de comillas/comentarios — a diferencia del splitter real de Go) y la detección de clause es "último keyword antes del cursor gana", con un ajuste puntual para `INSERT INTO t (col1, col2` (paréntesis sin cerrar después de la referencia de tabla → tratarlo como lista de columnas, no como querer renombrar la tabla). Los falsos positivos/negativos solo empeoran una sugerencia, nunca rompen la ejecución real — mismo umbral de tolerancia que el linter.

**El regex de `extractTableRefs` necesita una lista de "stopwords" de alias** (`WHERE`, `ON`, `JOIN`, `GROUP`, `SET`, etc.) — sin eso, `FROM users WHERE` capturaría "WHERE" como si fuera el alias de `users`. Cualquier keyword SQL nueva que pueda aparecer inmediatamente después de una referencia de tabla (p.ej. si se agrega soporte para alguna cláusula nueva) tiene que sumarse a `ALIAS_STOPWORDS`, o terminaría rompiendo la resolución de alias silenciosamente.

**Verificado standalone con un script de Node** (mismo patrón que `connStringParser.ts` — mirror en JS plano, sin tests nuevos) cubriendo los 13 casos del pedido original: FROM/INSERT INTO/UPDATE sugieren tablas; SELECT/WHERE/SET sugieren columnas acotadas a la tabla referenciada; `u.` resuelve el alias `u` de `FROM users u` a las columnas de `users`; `public.` resuelve el esquema `public` a sus tablas; `INSERT INTO users (id, ` cae a columnas de `users` pese a que el último keyword visto fue `INTO`; y aislar el statement correcto en un script multi-statement (`SELECT 1; SELECT * FROM users WHERE |`) antes de analizar el contexto.

# Tooltips obligatorios en todo control interactivo (post-lanzamiento)

Regla nueva en [.claude/rules/conventions.md](../../rules/conventions.md) (sección Frontend): todo `<button>`/`<select>`/checkbox no obvio lleva `title="..."` explicando qué hace pensando en un usuario que nunca usó la app — no una repetición del label. Motivada por un pedido explícito del usuario tras notar que ~40 controles del toolbar/diálogos no tenían ningún tooltip.

**Auditoría hecha con un script Python ad-hoc, no de memoria** — un primer intento de regex simple (`<button\b(.*?)>`) daba falsos positivos masivos porque `.*?` no-greedy corta en el primer `>` que encuentra, y arrow functions (`onClick={() => {...}}`) meten un `>` de sobra (el de `=>`) antes del cierre real del tag. La versión corregida busca el primer `>` que **no** está precedido por `=` — sigue fallando con comparaciones tipo `missing.length > 0` dentro de un `disabled={...}` (corta ahí también), así que el resultado de cualquier auditoría automática de JSX por regex debe tratarse como candidatos a revisar a mano, nunca como ground truth. Si se necesita re-auditar tras agregar componentes nuevos, no reusar el regex ciegamente — confirmar cada "MISSING" leyendo el bloque real.

**Tooltips condicionales nunca deben caer a `undefined` en la rama "no aplica"** — el botón "Test Connection" en `ConnectionDialog.tsx` tenía `title={cond ? '...' : undefined}`, que dejaba el botón sin tooltip en el caso común (no editando, password conocido). Corregido a que ambas ramas del ternario tengan texto — una explica el caso especial, la otra explica el comportamiento normal del botón.

**El botón "Regenerar CLAUDE.md" es el ejemplo de referencia para un tooltip que también comunica destructividad:** explica que sobreescribe archivos existentes y qué determina el alcance (schema seleccionado vs. toda la conexión) — no solo "regenera el CLAUDE.md". Cualquier acción irreversible/destructiva nueva (borrar, sobreescribir, desconectar con transacción abierta) debe seguir ese mismo nivel de detalle en el tooltip, no limitarse a nombrar la acción.

# Grilla de resultados: fix de overflow + selección de fila (post-lanzamiento)

**El fondo blanco "desbordado" en `ResultGrid.tsx` con muchas columnas era simplemente que el contenedor scrolleable nunca tenía `bg-*`** — cualquier área fuera de las celdas reales (tabla más angosta que el contenedor, o zona de scroll horizontal más allá de la última columna) mostraba el blanco default del documento en vez del tema oscuro. Fix: `bg-surface` en el div con `overflow-auto`, y el `<table>` pasó de `w-full` a `width: 100%` + `minWidth: table.getTotalSize()` (la suma real de anchos de columna, que crece si el usuario agranda una columna arrastrando el resize handle) para que el overflow horizontal ocurra de verdad en vez de comprimir columnas más allá de su `minSize`. Cualquier grilla nueva con columnas de ancho variable debe fijar el fondo del contenedor scrolleable explícitamente, nunca asumir que se hereda.

**Selección de fila = generar SQL, nunca edición inline con escritura automática a la DB.** El pedido original era "seleccionar la fila y poder exportarla/copiarla/editarla" — la interpretación deliberada de "editar" fue agregar `generateUpdateStatement` (`frontend/src/lib/sqlGenerate.ts`, mismo patrón que el `generateInsertStatements` ya existente) que arma un `UPDATE` con el `SET` de todas las columnas y un `WHERE` conservador que también matchea todas las columnas (comentario en el SQL generado recordando ajustar el `WHERE` a la primary key real), copiado al portapapeles para pegar y ejecutar a mano — nunca un cell editable que dispara un `UPDATE` real contra la base al perder foco. Mantiene la misma filosofía que el resto de la app: generar SQL, dejar que el usuario lo revise y lo corra explícitamente vía Ejecutar, nunca mutar datos por su cuenta. Cualquier feature futura de "editar datos" debe seguir este mismo patrón salvo que el usuario pida explícitamente edición inline con escritura automática — eso es una decisión de producto/seguridad mucho más grande (detección de PK, confirmación, riesgo de mutar producción sin querer) que amerita discutirse aparte, no asumirse.

# Selector de esquemas al crear una conexión nueva (post-lanzamiento)

**`ListSchemasForNewConnection` (app.go) es el equivalente de `TestConnection` para "listar esquemas antes de guardar":** ninguno de los dos tiene `connID` todavía (la conexión no existe en el vault hasta el submit), así que ambos arman la DSN desde `ConnectionInput` y abren una conexión de vida corta — `db.ListSchemasForDSN` sigue exactamente el patrón de `db.Ping` (`sql.Open` → operación → `defer conn.Close()`, nunca cachea en el `PoolManager`). Postgres-only, mismo motivo que el `ListSchemas(connID)` ya existente: Oracle no tiene un catálogo multi-schema equivalente en este modelo (`USER_*` views ya están acotadas al usuario conectado), así que el picker inline en `ConnectionDialog.tsx` solo se muestra para `dbType === 'postgres'` y `!editingId` (editar ya tiene su propio picker vía el ícono "esq" del sidebar → `SchemaPickerDialog`, no hacía falta duplicarlo ahí).

**El picker inline se dispara automáticamente después de un Test Connection exitoso, no con un botón aparte** — coincide con el flujo que pidió el usuario ("después de hacer test de conexión que te muestre los esquemas"). `SetConnectionSchemas` recién se llama en `doSave`, después de que `SaveConnection` devuelve el `connID` real — mismo criterio "todo marcado = sin restricción" que `SchemaPickerDialog` (si el usuario no desmarcó nada, no se persiste ninguna restricción).

# Backup/restore del vault requieren re-confirmar la clave maestra (post-lanzamiento)

**`Store.VerifyPassword(password)` (`backend/vault/store.go`) vs. `Unlock(password)`: mismo chequeo criptográfico, pero `VerifyPassword` nunca toca el `gate`.** `BackupVault` ahora pide la clave maestra otra vez aunque el vault ya esté desbloqueado — no es una verificación redundante, es una re-confirmación de identidad antes de escribir un archivo que puede terminar en otra máquina/USB/nube. Se agregó `VerifyPassword` en vez de reusar `Unlock` para dejar explícito que esto NO es un unlock (no debe re-setear el gate a la misma key ni sugerir que "desbloquear de nuevo" es lo que está pasando).

**`vault.VerifyBackupPassword(backupPath, password)` (`backend/vault/backup.go`) verifica la clave contra el `vault_meta.verifier` embebido EN EL BACKUP, no contra el vault local** — un backup puede restaurarse en una máquina distinta a la que lo generó, así que no hay ningún `vault_meta` local válido contra el cual comparar todavía. Extrae `vault.db`+`salt.bin` del zip a un directorio temporal descartable (`os.MkdirTemp` + `defer os.RemoveAll`), abre esa copia como una conexión SQLite aparte, deriva la key con el salt DEL BACKUP (nunca el salt local — son distintos si el backup viene de otra instalación), e intenta desencriptar el verifier. `App.RestoreVaultBackup` llama esto **antes** de tocar `vault.db`/`salt.bin` reales (antes de `a.vault.Close()`) — una clave incorrecta falla limpio sin dejar al usuario con un vault restaurado pero inaccesible.

**El campo de clave maestra de `UnlockScreen.tsx` es dual-propósito cuando `!isInitialized`:** el mismo `<input type="password">` sirve para "Crear vault" (clave nueva) o "Restaurar desde backup…" (clave que se verifica contra el backup elegido) — no se agregó un modal/campo separado para restore, se reusa el que ya está en pantalla. `restore()` valida que `password` no esté vacío localmente antes de llamar `onRestore(password)`, con un mensaje explícito ("Escribí la clave maestra del backup arriba") en vez de dejar que el backend rechace un string vacío sin contexto.

**`PasswordConfirmDialog.tsx` (nuevo, genérico) es el patrón a reusar para cualquier acción futura que necesite "re-confirmá tu clave maestra antes de esto"** — se auto-cierra si `onConfirm` no tira error, se queda abierto mostrando el error si tira (clave incorrecta, etc.) para reintentar sin repetir el flujo completo. Usado hoy solo por el backup (`Workspace.tsx`); restore no lo usa porque `UnlockScreen.tsx` ya tenía su propio campo de password en pantalla.

# Redis (path nativo paralelo, no `database/sql`)

**Decisión de arquitectura explícita, aprobada por el usuario — no asumida.** Redis rompe la regla dura "`database/sql` como única capa de acceso a datos" (`.claude/rules/technical.md` punto 2): no es relacional y `github.com/redis/go-redis/v9` no implementa las interfaces de `database/sql`. Forzarlo detrás de un driver `sql.DB` falso habría costado más código de adaptación que valor y habría perdido la semántica real de cada tipo (un hash no son filas de una tabla). En vez de eso, Redis vive en un path paralelo completo: `RedisPoolManager` (`backend/db/redis_pool.go`, sobre `redis.UniversalClient`) en vez de `PoolManager`, y `backend/redisquery` (mismo shape que `backend/query`: `Event`/`EmitFunc`/`HistorySink`/registro de `cancels`) en vez de un branch dentro de `query.Executor`. La excepción está documentada explícitamente en la regla, no implementada en silencio.

**`redis.NewUniversalClient(*redis.UniversalOptions)` unifica standalone/Cluster/Sentinel detrás de una sola interfaz** (`redis.UniversalClient`) — el tipo concreto que devuelve depende de los campos seteados: `MasterName` → cliente Sentinel-backed (mismo tipo concreto `*redis.Client` que standalone, solo que con reconexión automática al master vigente); `len(Addrs) >= 2` o `IsClusterMode: true` → `*redis.ClusterClient`; si no, `*redis.Client` simple. `Protocol: 2` (RESP2) fijado siempre en `redisUniversalOptionsFromDSN` — RESP3 colapsaría algunas respuestas (hashes, etc.) a shapes distintos sin ganar nada para este caso de uso.

**Un solo DSN string por conexión (igual que los otros 3 motores) codifica las 3 topologías** vía un `mode=` explícito en el query string (nunca inferido, a diferencia del truco de Oracle de inferir el modo por qué param está presente — Redis no tenía una señal igual de limpia). Ver `backend/db/redis.go`: standalone (`redis://[user:pass@]host:port[/db]?mode=standalone`), cluster (nodos completos en el query param `nodes=`, nunca en la autoridad de la URL — un host con comas rompe `url.Hostname()`/`Port()`), sentinel (`sentinels=` + `master=`, con `db` válido porque Sentinel entrega una conexión normal al master resuelto). **Decisión v1 documentada, no un límite oculto:** la autenticación contra los procesos Sentinel reusa el mismo `user:pass` de la URL (`SentinelUsername`/`SentinelPassword` = `Username`/`Password`) — go-redis soporta credenciales de Sentinel distintas, pero eso necesitaría su propio query param y una extensión de `export.RedactDSN` (que hoy solo toca `u.User`, y sigue funcionando sin cambios con el DSN actual).

**`SCAN` es el único comando que NO es uniforme entre topologías vía `redis.UniversalClient`** — es keyless, así que un `*redis.ClusterClient` no tiene con qué routearlo: llamarlo directo solo escanea el nodo que el cliente elija, no todo el keyspace. `backend/db/rediskeys.go`'s `ScanKeys` hace type-switch explícito: `*redis.ClusterClient` → `ForEachMaster` junta un `map[addr]*redis.Client` una vez por request, direcciones ordenadas con `sort.Strings` para que el cursor opaco (`"shardIdx:shardCursor"`) sea determinístico entre llamadas; cualquier otro tipo concreto (standalone y Sentinel-vía-`FailoverClient` comparten el mismo tipo concreto `*redis.Client`) → `Scan` simple con cursor `uint64`. Todos los demás comandos usados (`TYPE`/`TTL`/`GET`/`HSCAN`/`SSCAN`/`ZRANGE`/`LRANGE`/`XRANGE`) son key-scoped y sí rutean bien automáticamente vía la interfaz — no necesitan el mismo type-switch. **No verificado contra un Cluster real multi-shard en este entorno de desarrollo** — mismo precedente de honestidad que Oracle/PL-SQL en este mismo SKILL.

**Bug real evitado por leer la fuente en vez de asumir: `DurationCmd` de go-redis NO multiplica los sentinels de TTL por `time.Second`.** `TTL()` devuelve un `time.Duration`, y para los valores especiales de Redis (`-2` = la key no existe, `-1` = sin expiración) `command.go`'s `DurationCmd.readReply` los guarda tal cual en nanosegundos (`time.Duration(n)`), NO como `time.Duration(n) * precision` como hace con cualquier otro valor. Dividir por `time.Second` sin este chequeo especial (`int64(ttl / time.Second)`) colapsaría silenciosamente ambos sentinels a `0` — indistinguible de "expira en 0 segundos". `db.GetRedisKeyInfo` compara explícitamente contra `-1 * time.Nanosecond`/`-2 * time.Nanosecond` antes de hacer la división normal. Verificado leyendo el código fuente de `go-redis` (`command.go`), no asumido de la documentación.

**Paginación nunca "trae todo de una sola vez", ni siquiera para un valor que se ve chico** (`backend/db/rediskeys.go`'s `GetRedisValue`): hash/set usan el cursor propio de `HSCAN`/`SSCAN`; list/zset usan una ventana `offset`/`count` (con un truco de reuso: el campo `Cursor` de la respuesta duplica como "próximo offset" para estos dos tipos en vez of un cursor real, ya que `LRANGE`/`ZRANGE` no tienen cursor propio — el frontend (`RedisValueInspector.tsx`) sabe cuál de los dos interpretar según `type`); stream usa `XRANGE` con un ID de inicio exclusivo (`"(" + lastID`) sobre el último ID visto.

**Fuera de alcance v1, documentado explícitamente — no un límite silencioso:**
- **MULTI/EXEC/WATCH/DISCARD**: el executor despacha cada comando con `client.Do()` contra lo que el pool le dé — sin pinear una conexión (como si hace `query.Executor.txConn` para SQL, vía `client.Conn()`), un `MULTI`/`EXEC` pegado en el editor podría correr en conexiones distintas y ejecutar mal EN SILENCIO. Se prefirió no ofrecerlos (ni como snippet en `redisLanguage.ts`) antes que soportarlos mal.
- **SUBSCRIBE/PSUBSCRIBE**: streams de larga duración, no encajan en el modelo comando→resultado único del editor de comandos.
- **EXPLAIN/DDL/generación de CLAUDE.md**: no existen esos conceptos en Redis — quedan fuera del `switch dbType` correspondiente en `app.go` sin agregar un caso nuevo.

**`backend/redisquery.Executor.run` replica el mismo bug real ya documentado en `query.Executor.run`: `break`, no `continue`, al detectar cancelación** — un `continue` emitiría un evento "cancelled" fantasma por cada comando restante del script en vez de simplemente dejar de correrlos.

**Frontend: `RedisResultView.tsx` es una transcripción tipo redis-cli (un bloque por comando corrido), no una grilla tabular** — se abandonó la idea original de reusar `ResultGrid` directamente al notar que ese componente trae acciones SQL-específicas ("copiar como INSERT/UPDATE") que no tienen sentido para un hash o una lista de Redis. `ExportMenu` (CSV/JSON/XLSX, sin las acciones SQL) sigue reusándose tal cual por comando, alimentado por `lib/redisResultToTable.ts` — un adapter puro que aplana `resultKind`/`result` a `columns`/`rows` en el frontend, así que `backend/export`'s `WriteCSV`/`WriteJSON`/`WriteXLSX` no necesitaron ningún cambio.

**`RedisKeyTree.tsx` reemplaza el árbol de tablas dentro de `ConnectionTree.tsx` (no reemplaza `ConnectionTree` entero)** — la lista de conexiones, su ícono, expandir/colapsar, editar/exportar config/desconectar siguen siendo agnósticas al motor; solo el cuerpo expandido de una conexión Redis (antes: `metadata.tables`) se reemplaza por el árbol de keys paginado (SCAN, nunca `KEYS *`). Reusa el prop `reloadToken` que `ConnectionTree` ya tenía para refrescar la lista de conexiones — también fuerza un refresh del árbol de keys después de un delete hecho desde `RedisValueInspector.tsx` (un componente distinto, sin estado compartido más allá de ese token).

# MongoDB (segundo path nativo paralelo, no `database/sql`)

**Decisión de arquitectura explícita, aprobada por el usuario — la SEGUNDA excepción a la regla dura de `database/sql` (`.claude/rules/technical.md` punto 2), igual que Redis.** MongoDB es documental, no relacional, y `go.mongodb.org/mongo-driver/v2` (el driver oficial, la librería más adecuada, pure-Go sin cgo) no implementa las interfaces de `database/sql`. Vive en un path paralelo completo espejando exactamente al de Redis: `MongoPoolManager` (`backend/db/mongo_pool.go`, sobre `*mongo.Client`) en vez de `PoolManager`, y `backend/mongoquery` (mismo shape que `backend/query`/`backend/redisquery`: `Event`/`EmitFunc`/`HistorySink`/registro de `cancels`, y el mismo bug-precedent `break`-no-`continue` al cancelar) en vez de un branch dentro de `query.Executor`. **Ruteo por bindings separados que el frontend elige por `dbType`, NUNCA un `switch dbType` server-side dentro de `ExecuteQuery`** — `ExecuteMongoQuery`/`CancelMongoQuery` conviven con `ExecuteQuery`/`ExecuteRedisCommand`, igual que Redis. `app.go` cierra `mongoPools` en Disconnect/Delete/Update/shutdown junto a los otros managers (todos no-op para el motor que no poseen).

**Connector (`backend/db/mongo.go`):** DSN URL-shaped que el driver consume vía `ApplyURI` — `mongodb://user:pass@host:port[,host2...]/db?opts` (standard) o `mongodb+srv://host/db?opts` (SRV/Atlas). **A diferencia de Redis, el modo NO viaja en un query param** — el driver rechaza opciones desconocidas, así que se infiere del scheme (`mongodb` vs `mongodb+srv`) y de la forma de la autoridad (una autoridad con comas = replica set → `params["hosts"]`), como hace Oracle al inferir su modo. Solo se emiten opciones válidas de Mongo (`replicaSet`/`authSource`/`tls`). `export.RedactDSN` funciona sin cambios (credenciales en `u.User`). Auth: user/password (SQL auth); todo opcional (muchos Mongo corren sin auth, como Redis). **`ParseDSN` lee la autoridad de `u.Host` crudo, NO `u.Hostname()`/`u.Port()`** — esos parten mal una autoridad multi-host `h1:27017,h2:27017`.

**Editor: lenguaje mongosh hecho a mano** (`frontend/src/codemirror/mongoLanguage.ts`, `TabLanguage='mongosh'`), mismo criterio que redis-cli (no hay paquete oficial de CodeMirror): `StreamLanguage` + `CompletionSource` (métodos tras `db.<coll>.`, colecciones tras `db.` desde `mongoCollectionsStore.ts`, `$operadores`) + `hoverTooltip`. El backend parsea `db.<coll>.<método>(args)[.sort()/.limit()/...]` con `parser.go` (consume un statement balanceado a la vez, robusto a args multi-línea; NO parte por `;`/newline ingenuo). **Parser lenient de argumentos (`extjson.go`, decisión del usuario):** convierte sintaxis mongosh real a Extended JSON estricto — claves sin comillas, comillas simples, comentarios, comas colgantes, y helpers `ObjectId()/ISODate()/Date()/new Date()/NumberLong()/NumberDecimal()/NumberInt()/UUID()`. Es un recursivo-descendente acotado, NO un motor JS (mismo argumento que `splitter.go` vs. una gramática completa) — lo que no reconoce es un error claro, nunca un mis-parse silencioso. El panel de edición de documentos usa Extended JSON ESTRICTO (los valores vienen ya de la DB), solo el editor usa el lenient.

**Métodos soportados** (`mongoquery/executor.go` dispatch): find/findOne/aggregate/countDocuments/distinct/estimatedDocumentCount/insertOne/insertMany/updateOne/updateMany/replaceOne/deleteOne/deleteMany/createIndex/dropIndex/getIndexes, más los modificadores de cursor sort/limit/skip/projection/count. **`find` sin `.limit()` explícito aplica un límite por defecto de 50** (protege la UI; documentado en el summary del resultado). Todos los resultados se emiten como documentos Extended JSON (`Event.Documents []string`) + un `summary`, así que el frontend los renderiza siempre como JSON con color sin importar si fue un find, un insert (devuelve `{acknowledged, insertedId}`) o un update (`{matchedCount, modifiedCount, ...}`). **`db` en mongosh apunta a la base ACTIVA que el frontend pasa explícitamente** (`ExecuteMongoQuery(connID, queryID, database, cmd)`) — Mongo navega muchas bases, así que no hay "base seleccionada" server-side; `Workspace.tsx` cachea la base activa por conexión (`mongoDbByConn`, mismo patrón que `activeSchemaByConn`), sembrada del default del DSN (`GetMongoDefaultDatabase`) y actualizada al expandir una base o abrir una colección en el árbol.

**Frontend (espeja Redis, componentes en `frontend/src/components/mongo/` y `results/`):** `MongoResultView.tsx` (transcripción, un bloque por comando) usa `results/JsonView.tsx` — un visor JSON recursivo, colapsable y con color (la "sección de resultado nueva JSON con formato color" pedida), que renderiza los wrappers Extended JSON (`{$oid}`→`ObjectId("…")`, `{$date}`→`ISODate("…")`, etc.) como átomos tipados igual que Compass. `MongoCollectionTree.tsx` reemplaza el árbol de tablas dentro de `ConnectionTree` para `dbType==='mongodb'` — **multi-DB: bases → colecciones (+conteo estimado) → índices** (decisión del usuario, matchea Compass). `MongoBrowserTab.tsx` (kind `'mongo-browser'`) es un tab de dos paneles: árbol de colecciones + `MongoDocumentPanel.tsx` (documentos paginados, editar uno como JSON → `ReplaceMongoDocument` por `_id`, eliminar → `DeleteMongoDocument`). `MongoFindWizard.tsx` es el asistente visual de `find` (campo/operador/valor → genera `db.coll.find({...})`, con auto-detección de 24-hex → `ObjectId("…")`) para quien no sabe MQL. `lib/mongoResultToTable.ts` aplana documentos a `columns`/`rows` para reusar `ExportMenu` (CSV/JSON/XLSX) sin tocar `backend/export`. Lint (`lib/mongoLinter.ts`): `deleteMany/updateMany` con filtro `{}` y `drop()` son `blocking` (confirmación previa), el análogo de `DELETE/UPDATE` sin `WHERE`.

**Verificación: SÍ se probó el backend completo contra un MongoDB real** (contenedor Docker efímero `mongo:7`, descartado al terminar) vía un script Go de un solo uso que ejercitó todo el pipeline: BuildDSN/ParseDSN roundtrip, `RedactDSN`, `MongoPoolManager.Open`/Ping, y el `mongoquery.Executor` corriendo insertMany/find (con filtro lenient `ObjectId()`/claves sin comillas/comillas simples/`$gt`)/find con `.sort().limit()`/findOne/countDocuments/distinct/aggregate/updateMany/createIndex/getIndexes/deleteOne/error de método desconocido/script multi-comando, más `db.ListMongoDatabases`/`ListMongoCollections`/`GetMongoIndexes` y browse/edición de documentos (`ListMongoDocuments`/`CountMongoDocuments`/`ReplaceMongoDocument`/`DeleteMongoDocument`) — todos OK. El flujo completo vía la UI de Wails (árbol, wizard, edición en el panel) sigue sin probarse automáticamente en este entorno (requiere interacción), pero el backend que lo alimenta está verificado end-to-end. Binario: 48MB (+4MB, ver `.claude/rules/technical.md` punto 8).

# Pestañas del editor con conexión + lenguaje propios (post-Redis)

**Decisión de arquitectura confirmada explícitamente con el usuario, no asumida:** vincular una pestaña a una conexión es SIEMPRE un acto explícito vía el selector propio de la pestaña (`EditorTabs.tsx`, ícono a la izquierda del título) — clickear una conexión en el sidebar NUNCA vincula la pestaña activa. Esto significa que `selected` (`Workspace.tsx`) y `activeTabConnection` son dos piezas de estado genuinamente independientes: `selected` es solo "qué conexión tiene expandida el árbol del sidebar" (navegación de tablas/keys), `activeTabConnection` es "a qué conexión corre/autocompleta la pestaña activa" (`connections.find(c => c.id === activeTabData?.connId)`). Pueden coincidir o no en cualquier momento — el toolbar, el editor, y el autocomplete siguen a `activeTabConnection`; el árbol de tablas/keys del sidebar sigue a `selected`. Meterle a uno un dato derivado del otro reintroduciría exactamente el acoplamiento que se pidió romper.

**Metadata de schema y "esquema activo" cacheados por `connId`, no como un único `useState`** (`metadataByConn`/`activeSchemaByConn` en `Workspace.tsx`) — necesario porque el sidebar y el editor pueden estar mirando conexiones distintas a la vez; sin el cache por conexión, cambiar de pestaña forzaría un refetch de metadata cada vez incluso volviendo a una conexión ya cargada, y el sidebar y el editor se pisarían el `activeSchema` mutuamente. Dos efectos independientes (`ensureMetadata` por `selected.id` y por `activeTabConnection.id`) alimentan el mismo cache — si coinciden, cache hit, sin fetch duplicado.

**Un solo editor montado, con un modelo por pestaña, reemplaza el diseño anterior de dos componentes que se montaban/desmontaban según el motor.** Con conexión-por-pestaña, cambiar entre una pestaña SQL y una Redis pasó a ser rutinario (es el punto central de la feature) — remontar el editor completo en cada cambio de tipo hubiera reseteado scroll/cursor/undo-stack en cada switch, exactamente lo contrario de la fluidez pedida. Esto se implementó originalmente sobre Monaco (`editor.setModel(model)` entre `monaco.editor.ITextModel`s cacheados en un `Map<tabId, ITextModel>`) y hoy es CodeMirror (`view.setState(state)` entre `EditorState`s cacheados en un `Map<tabId, EditorState>`, `CodeMirrorTabbedEditor.tsx`) — mismo diseño, ver "Patrones de CodeMirror" arriba para el detalle actual. **Gotcha real evitado a propósito, válido en ambas implementaciones:** el efecto que hace el swap de modelo/estado está keyeado SOLO en `activeTabId` (leyendo `tabs` vía un ref, no como dependencia directa) — si dependiera de `tabs` completo, se re-ejecutaría en cada keystroke (la referencia de `tabs` cambia con cada cambio de contenido) y resetearía scroll/cursor mientras se tipea.

**`backend/vault/settings_repo.go`'s `open_tabs` pasó de persistir `[]string` (solo paths) a `[]OpenTabInfo` (path + connId + language opcionales) SIN migración SQL nueva** — la columna ya era `TEXT` libre con JSON adentro (migración v3), así que solo cambió qué struct Go la (de)serializa. **Retrocompatibilidad obligatoria, no opcional** (`vault.db` ya instalado en máquinas reales, regla 13 de `.claude/rules/technical.md`): `GetSettings` intenta deserializar como `[]OpenTabInfo` primero; si falla, cae a `[]string` legado y envuelve cada path como `OpenTabInfo{Path: p}`. Verificado con un script efímero sandboxeado (`HOME=$(mktemp -d)`, patrón ya establecido para tocar vault/crypto) que escribe el shape legado a mano en `settings.open_tabs` y confirma que `GetSettings` lo lee bien, más el round-trip del shape nuevo — ambos casos pasaron.

**Un tab bindeado a una conexión eliminada se resuelve solo, sin código especial:** `activeTabConnection` es `connections.find(...) ?? null` — si la conexión vinculada ya no existe (borrada mientras la app estaba cerrada, o recién eliminada con el nuevo botón "Eliminar conexión"), el `.find()` simplemente no encuentra nada y la pestaña se ve como no-vinculada. `deleteConnection` (`Workspace.tsx`) además limpia explícitamente `tab.connId` a `null` en cualquier pestaña que apuntara a esa conexión (no imprescindible para la UI ya que el `.find()` ya degradaría bien, pero evita persistir un `connId` fantasma en el próximo `SetOpenTabs`). El archivo/contenido de la pestaña nunca se toca — solo el vínculo.

**"Eliminar conexión" no existía como acción de UI antes de esta feature** (`App.DeleteConnection` del backend nunca se invocaba desde el frontend) — se agregó un botón nuevo en `ConnectionTree.tsx` (ícono papelera, junto a desconectar), gatillado por `ConfirmDialog` (nunca `window.confirm`, ver `.claude/rules/conventions.md`) porque es destructivo y permanente.

**Intellisense de Redis: se evaluó explícitamente sumar una librería (pedido del usuario) y se descartó** — la única con metadata real de comandos (`redis-commands` de npm) solo trae `arity`/`flags`/posición de key para ruteo de Cluster, no nombres de argumento ni docs legibles; la fuente realmente rica (`src/commands/*.json` del repo de Redis) no se distribuye como paquete. Se extendió a mano el spec por comando ya existente en `redisLanguage.ts` con un flag `firstArgIsKey` (true para GET/SET/DEL/HGET/etc., false para PING/SELECT/FLUSHDB/FLUSHALL) — sin dependencia nueva, mismo criterio que el resto del proyecto (SKILL ya documenta esta misma decisión para SQL). `redisKeysStore.ts` (nuevo, mismo mutable-holder pattern que `metadataStore.ts`) alimenta la sugerencia de keys conocidas en la posición del primer argumento; `RedisKeyTree.tsx` solo empuja sus keys ahí cuando su `connId` coincide con `activeTabConnection` (prop `isActiveTabConnection`) — si no, una conexión distinta expandida en el sidebar contaminaría las sugerencias de una pestaña que apunta a otra. Se agregó también un `registerHoverProvider('redis-cli', ...)` que no existía en absoluto antes (mismo shape que el hover de SQL).

**Fix real en el hover de SQL:** antes siempre mostraba el tooltip de la PRIMERA tabla en `metadata.tables` que tuviera una columna con ese nombre, sin usar el alias del cursor — un JOIN entre dos tablas con una columna repetida podía mostrar el tooltip de la tabla equivocada. Ahora reusa `resolveDotPrefix`/`extractTableRefs` de `sqlContext.ts` (la misma máquina que `completionProvider.ts` ya usaba) para priorizar las tablas efectivamente referenciadas por el FROM/JOIN de la sentencia actual antes de caer al resto del schema.

**`frontend/src/lib/likePattern.ts` (nuevo) unifica la sintaxis de búsqueda entre el filtro de tablas (SQL, `ConnectionTree.tsx`) y el filtro de keys (Redis, `RedisKeyTree.tsx`)** — un término plano se auto-envuelve para "contiene" (comportamiento que el filtro SQL ya tenía implícito, y que el filtro de Redis no tenía — antes exigía tipear el glob de SCAN a mano); `%`/`_` (convención SQL LIKE) se traducen a regex o a glob de Redis según el destino; `*`/`?`/`[...]` ya tipeados se respetan tal cual, nunca se traducen dos veces.

**Verificación:** `go build`/`go vet`/`pnpm build` pasan limpios. **Sí se probó contra un Redis standalone real** (contenedor Docker efímero, descartado al terminar) vía un script Go de un solo uso que ejercitó el pipeline completo: `BuildDSN`/`ParseDSN` roundtrip, `PingRedisDSN`, `RedisPoolManager.Open`, y `redisquery.Executor` corriendo SET/GET (incluyendo un GET a una key inexistente → `resultKind: "nil"`, no error)/HSET/HGETALL/LPUSH/LRANGE/SADD/SMEMBERS/ZADD/ZRANGE WITHSCORES/EXPIRE/TTL/INCR/DEL/PING, más `db.ScanKeys`/`GetRedisKeyInfo`/`GetRedisValue` sobre las keys creadas — todo con resultados correctos. Los round-trips de DSN para Cluster/Sentinel/TLS también se verificaron (sin infraestructura real detrás, solo el parsing). Cluster/Sentinel contra nodos reales, y el flujo completo vía la UI de Wails, siguen sin probarse en este entorno de desarrollo — mismo precedente de honestidad que Oracle/PL-SQL.

# Autocompletado, auto-guardado y terminal SSH (post-MongoDB)

**Funciones SQL en el autocompletado (`frontend/src/codemirror/sqlFunctions.ts`):** cuarta fuente `autocomplete` aditiva registrada en `sqlLanguageExtension` (`sqlSchema.ts`), junto a la schema-aware, los snippets y el `keywordCompletionSource` de lang-sql. Cierra el hueco de que lang-sql solo completa keywords, no funciones incorporadas. Lista común + por dialecto (`DIALECT_FUNCS[dbType]`); reusa `detectClause` (exportada de `sqlSchema.ts`, mismo import que `sqlSnippets.ts`) para NO ofrecer funciones justo después de FROM/JOIN. Al agregar un motor nuevo, sumar su entrada a `DIALECT_FUNCS`.

**Auto-guardado de editores (setting `auto_save_enabled`/`auto_save_interval_seconds`, migración v16):** el patrón de settings es idéntico al de auto-backup (columna en `settings_repo.go` + `SetX` validado + binding en `app.go`), PERO **no hay scheduler en Go** — el timer vive enteramente en el frontend (`Workspace.tsx`, un `setInterval` en un `useEffect` gateado por el toggle). Solo guarda pestañas con `dirty && path && kind==='editor'` vía `SaveSQLFile` (nunca `SaveSQLFileAs`, que abriría un diálogo nativo — mal en un timer); las scratch tabs sin archivo se ignoran a propósito. Lee `tabsRef.current` (no `tabs`) para que el intervalo no se re-suscriba en cada tecla.

**Sugerencia fantasma en la terminal SSH (`frontend/src/lib/sshLineModel.ts` + `SshTerminalTab.tsx`):** la terminal manda cada tecla cruda al PTY con ECHO del server, así que NO hay modelo de línea — `SshLineModel` reconstruye la línea actual observando el mismo `onData` (printables/backspace/Ctrl-A/E/U/K/flechas ←→). Es **deliberadamente conservador**: Tab (completado remoto), ↑/↓ (recall de historial) o cualquier escape desconocido ponen `desynced=true`, que suprime la sugerencia hasta el próximo Enter — una sugerencia equivocada es peor que ninguna. **Clave: una línea `desynced` NO se registra en el historial al commitear** (su reconstrucción es poco confiable; registrarla envenenaría sugerencias futuras). Historial solo en memoria de la sesión, nunca persistido (los comandos SSH pueden tener secretos). La aceptación (Tab o → con ghost presente) manda el sufijo por `WriteSSHTerminal` y avanza el modelo con `accept()`; sin ghost, Tab/→ pasan al shell remoto igual. El overlay gris se posiciona leyendo `term.buffer.active.cursorX/cursorY` (API pública) × el tamaño de celda (`.xterm-screen` rect / cols·rows), reposicionado tras cada echo del server (callback de `term.write`, porque el cursor recién se mueve cuando el PTY hace echo). Verificado con un mirror-test en JS plano de `process`/`suggestion`/`commit` (10 casos: sufijo, backspace, Tab-desync, no-registrar-desynced, resync tras Enter, accept, cursor-no-al-final). El flujo visual real en la terminal Wails no se probó automáticamente (requiere interacción).

**UI de MongoDB (mejoras):** el editor de un documento en `MongoDocumentPanel` usa `mongo/JsonEditor.tsx` — un CodeMirror mínimo (controlado, mount-once) con `@codemirror/lang-json` + `jsonParseLinter` (valida en vivo, el botón Guardar se deshabilita si el JSON es inválido vía `onValidityChange`); su chrome se estiliza con las CSS-vars de diseño (`--color-surface-container-low`, etc.) para adaptarse a claro/oscuro sin prop de tema. `MongoResultView` suma un toggle JSON↔tabla (la tabla reusa `mongoResultToTable`) + un buscador que filtra documentos por texto sobre su ExtJSON. El selector de base activa vive en el toolbar de `Workspace.tsx` (gateado por `isMongoActive`), alimentado por `mongoDatabasesByConn` (fetch de `ListMongoDatabases` al activar la conexión) y escribe en `mongoDbByConn` vía `selectMongoDatabase` — el mismo estado que el árbol actualiza al expandir una base.
