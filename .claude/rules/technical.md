# Regla técnica — restricciones duras de mini-tools

Estas restricciones son no negociables; cualquier cambio que las viole necesita discutirse explícitamente con el usuario antes de implementarse, no asumirse.

1. **Sin cgo.** Toda dependencia de acceso a datos debe ser pure-Go. `modernc.org/sqlite` sí, SQLCipher o cualquier driver que requiera cgo no. Esto es lo que permite compilar sin toolchain de C y mantener el binario portable.
2. **`database/sql` como única capa de acceso a datos — con una excepción explícita y documentada: Redis.** Los tres motores relacionales (Oracle vía `go-ora/v2`, Postgres vía `pgx/v5/stdlib`, SQLite vía `modernc.org/sqlite`) se registran como drivers `database/sql`. Prohibido usar `sqlx` o el SDK nativo de un motor relacional directamente fuera de `backend/db/`. **Redis es la única excepción, aprobada explícitamente (no asumida):** no es relacional y `github.com/redis/go-redis/v9` no implementa las interfaces de `database/sql`, así que corre por un path nativo paralelo — `RedisPoolManager` (`backend/db/redis_pool.go`) sobre `redis.UniversalClient` en vez de `PoolManager`/`*sql.DB`, y `backend/redisquery` en vez de `backend/query`. Ver [.claude/skills/mini-tools-patterns/SKILL.md](../skills/mini-tools-patterns/SKILL.md) sección "Redis" para el diseño completo. Cualquier motor nuevo que sí sea relacional sigue la regla original sin excepción.
3. **Vault: cifrado a nivel de columna, no de archivo.** `vault.db` es una SQLite normal; solo la columna `encrypted_dsn` (+ `nonce`) va cifrada con AES-256-GCM. Cifrar el archivo completo (tipo SQLCipher) está descartado porque requiere cgo.
4. **Derivación de clave: Argon2id, salt fijo por instalación.** La clave maestra nunca se persiste en ningún lado; el salt sí se persiste (en `appdata/salt.bin`) pero por separado del vault.
5. **Gate del vault aplicado server-side.** Cada método bindeado en `App` que toque datos de conexión o del vault debe verificar el flag `unlocked` de `vaultgate` — la UI puede reforzarlo mostrando la pantalla de unlock, pero la verificación real vive en Go. **Única excepción deliberada:** `GetSettings`/`SetTheme` (Fase 10) — aunque viven en el mismo `vault.db`, la tabla `settings` no tiene nada sensible (nunca DSN, nunca datos de fila), así que gatearla detrás de la clave maestra solo causaría un flash de tema incorrecto en la pantalla de desbloqueo sin ganar seguridad real. Cualquier dato nuevo que sí sea sensible debe seguir la regla general sin excepción.
6. **CodeMirror 6, sin CDN, sin worker de Vite.** Editor único vía `@codemirror/{state,view,commands,language,autocomplete,lint,search,lang-sql}` + el meta-paquete `codemirror` (`basicSetup`) + `@uiw/codemirror-theme-*` para los temas — todo bundleado localmente, nada se descarga en runtime. SQL usa `@codemirror/lang-sql` con dialectos reales (`PostgreSQL`/`SQLite`/`PLSQL` según `dbType`) en vez de una lista de keywords hecha a mano; Redis no tiene paquete oficial, se resuelve a mano en `frontend/src/codemirror/redisLanguage.ts` (`StreamLanguage` + `CompletionSource` + `hoverTooltip`, puerto directo de la versión Monaco). A diferencia de Monaco, CodeMirror no tiene providers globales/singleton — cada pestaña tiene su propio `EditorState` con su propio schema/dialecto/tema vía `Compartment` (`frontend/src/components/editor/CodeMirrorTabbedEditor.tsx`), así que no hace falta un mutable-holder global tipo el retirado `metadataStore.ts`/`activeDbTypeStore.ts`. Tampoco hace falta cablear un web worker de Vite a mano — CM6 no lo necesita, a diferencia de `monaco-editor/esm/vs/editor/editor.worker`.
7. **Sin librería de parsing SQL.** La detección de PL/SQL, el split de statements y el linter básico son hechos a mano (`backend/query/detect.go`, `splitter.go`, `frontend/src/lib/linter.ts`). Librerías tipo vitess sqlparser / pg_query_go / gramáticas ANTLR quedan descartadas por peso y por no cubrir Oracle+Postgres+SQLite a la vez.
8. **Binario de producción <60MB** (techo revisado cuatro veces — Fase 4, Fase 6, el agregado de Redis, y una revisión explícita del usuario de 45MB a 60MB tras esa medición, ver nota abajo. El target original de <20MB del spec no es alcanzable con los motores nativos + Monaco en un solo binario). Verificar con `wails build` + `ls -lh build/bin/*` (macOS) en cada fase que añade una dependencia grande (`excelize`, etc.). `wails build` ya corre con `-ldflags "-w -s"` por default (confirmado con `-v 2`) — no hay margen extra ahí, no hace falta pasarlo a mano. Fase 1 (scaffold): ~7.6MB. Fase 2 (+ `modernc.org/sqlite` + `golang.org/x/crypto/argon2`): ~12MB. Fase 3: sigue en ~12MB. Fase 4 (+ `pgx/v5/stdlib` + `go-ora/v2`): ~31MB — Oracle agrega ~15MB porque `go-ora` usa `crypto/tls` en su código core (soporte TCPS, no opcional) y desde Go 1.24 `crypto/tls` arrastra el módulo FIPS 140-3 completo; Postgres agrega ~6MB. Fase 6 (+ Monaco recortado a SQL): ~35MB — el core de Monaco (`editor.api`, sin ningún lenguaje extra) pesa ~3.9MB minificado por sí solo, es inherente a la librería, no hay una versión "core" más chica. Fase 7 (`excelize` para export XLSX) apenas movió la aguja. Fase 10 (con theming/lazy-load/fixes de seguridad, sin dependencias nuevas): 37MB. **Agregado de Redis (`github.com/redis/go-redis/v9`): 44MB — +7MB, más de lo esperado para un cliente pure-Go sin cgo** (la sospecha, no confirmada con un profiler: go-redis expone cientos de métodos tipados por comando sobre `Cmdable`, y como se invocan varios de esos métodos genéricamente vía la interfaz `redis.UniversalClient`/`Cmdable`, el linker no puede eliminar por dead-code el resto de la superficie de comandos no usada dentro del mismo paquete). Esto dejaba el binario a 1MB del techo original de 45MB — el usuario decidió subir el techo a **60MB** en vez de perseguir ese margen, dejando ~16MB de aire cómodo sobre los 44MB actuales. **Migración Monaco → CodeMirror 6: 41MB — una baja de ~3MB** (medido con `wails build` tras retirar `monaco-editor` y sumar `@codemirror/*` + `codemirror` + `@uiw/codemirror-theme-*`; CodeMirror 6 resultó más liviano que Monaco recortado a SQL pese a sumar autocompletado por esquema, dialectos SQL reales, lint y search que Monaco no traía). Cualquier dependencia grande nueva de acá en adelante sigue requiriendo medir el delta real antes de mergear (la disciplina no cambia, solo el número); si se vuelve a acercar al nuevo techo, las mismas opciones de antes siguen disponibles: evaluar `wails build -upx` (compresión UPX del binario final, si está instalado — la única palanca de compresión que expone Wails, nunca usada hasta ahora en este proyecto ni verificada acá; la nota de fases anteriores que mencionaba "la palanca de gzip" se refería a esto, nombrado de forma imprecisa) o discutir con el usuario otra revisión del techo.
9. **El frontend nunca ve un DSN ni un password.** Solo IDs de conexión opacos cruzan el binding Go↔React.
10. **Nunca loguear un DSN ni resultados de queries**, tampoco en modo debug.
11. **pnpm únicamente** para el frontend; sin Node como runtime en producción.
12. **Sin Electron, sin frameworks extra sobre Wails.** Dependencias mínimas en el backend Go — no agregar un framework HTTP, un ORM, ni una capa de UI nativa adicional solo porque "podría ser útil después".
13. **`vault.db` ya está instalado y en uso en varias máquinas reales — cualquier columna o tabla nueva se agrega con una migración (`backend/vault/migrations.go`), nunca borrando/recreando la base.** Este vault no es un entorno de desarrollo descartable: tiene conexiones guardadas, historial de queries, archivos recientes, tema, layout — borrar `vault.db` (o sugerir "borralo y que se regenere solo") destruye eso en cada máquina donde ya está instalado, no solo en la de desarrollo. Migraciones: solo aditivas, `vault_meta.verifier`/`verifier_nonce` intocables. Ver [.claude/specs/vault-migrations.md](../specs/vault-migrations.md) para el diseño completo. Cualquier entrada en `migrations.go` debe ser `CREATE TABLE IF NOT EXISTS` o `ALTER TABLE ... ADD COLUMN ... DEFAULT ...` — nunca `DELETE`/`DROP TABLE`/mutar filas existentes, y nunca tocar la fila de `vault_meta` (el verificador criptográfico). Es la misma clase de restricción no negociable que los puntos 3/4 sobre cripto — una migración que rompa el desbloqueo o pierda datos de un usuario existente es peor que no tener la migración. Esto aplica incluso durante desarrollo activo: el vault local de la máquina de desarrollo se trata con el mismo cuidado que el de un usuario real, nunca se borra como atajo para "empezar de cero" — si hace falta una base limpia para probar algo, se usa un script efímero sandboxeado (`HOME=$(mktemp -d)`), nunca tocar `~/Library/Application Support/mini-tools` directo.
